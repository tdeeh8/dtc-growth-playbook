# Audit Report Template

This is the canonical structure for all audit reports generated by the synthesizer. Follow this section order exactly. Omit sections only when explicitly noted (e.g., Cross-Channel Diagnosis in single-platform mode).

**Output format:** Markdown by default. DOCX only if user explicitly requests it.

---

## Filename Convention

- Markdown: `{Client}_audit_report_{YYYY-MM-DD}.md`
- DOCX: `{Client}_audit_report_{YYYY-MM-DD}.docx`
- Location (Disruptive): `{project_dir}/`
- Location ({client_name}): `{project_dir}/`

---

## Report Structure

### Header Block

```markdown
# {Client Name} — Growth Audit Report

**Date:** {Report generation date}
**Prepared by:** Tanner Hill, {Department context}
**Platforms audited:** {list of platforms with evidence files}
**Date range:** {earliest start date} to {latest end date across all evidence}
**Report type:** {Single-Platform / Cross-Channel Synthesis}
```

---

### Section 1: Executive Summary

**Length:** 2-3 paragraphs. No bullet points. Business-owner readable — no jargon.

**Paragraph 1:** State the headline finding. What is the single biggest thing affecting this business's growth? Lead with the answer, not the process. Example: "Kodiak Leather is leaving an estimated $17K-$29K in revenue on the table because its two best-performing campaigns are budget-constrained while exceeding their ROAS targets."

**Paragraph 2:** Supporting context. What other significant findings support or complicate the headline? Mention 2-3 secondary findings. Reference the profitability picture if Shopify data is available.

**Paragraph 3:** What to do about it. Summarize the top 2-3 recommended actions in plain language. Reference the Tiered Roadmap for implementation order.

**Rules:**
- No metrics in the exec summary unless they're essential to the headline (e.g., "3.91x ROAS vs. 3.3x target"). Keep it to 2-3 numbers max.
- Write for a business owner, not a media buyer. "You're paying for clicks that aren't turning into sales" not "CTR/CVR ratio indicates post-click conversion inefficiency."
- Every claim traces to a finding or evidence — but don't cite sources inline here. The rest of the report handles that.

### Section 2: What Was Audited

```markdown
## What Was Audited

| Platform | Depth | Date Range | Evidence File |
|----------|-------|------------|---------------|
| {platform} | {deep/standard} | {start} – {end} | {filename} |
```

One row per platform with an evidence file. Include the evidence filename for traceability.

If a manifest exists, also note:
- Audit type (full / platform-specific)
- AOV tier
- Who requested the audit and any specific focus areas

---

### Section 3: Accessible vs. Missing

```markdown
## What This Report Covers — and What It Can't

**Platforms analyzed:** {list}

**Not included in this audit:**
- {Platform} — {reason: SKIPPED / NOT_STARTED / no access}

**What this means for the analysis:**
- {specific limitation, e.g., "Without Shopify data, profitability figures use platform-reported revenue (typically inflated 20-50%). Treat margin calculations as directional only."}
- {specific limitation}
```

Be explicit about what's missing and how it limits the analysis. Don't bury this — stakeholders need to know the boundaries of the report's claims.

### Section 4: Tracking Health

**Include this section only if tracking issues were detected in any evidence file.**

If no tracking issues exist across any platform, omit this section entirely.

```markdown
## Tracking Health

{If high-severity issues exist, lead with:}
> ⚠️ **Tracking issues detected.** Performance metrics in this report may be inflated or understated. Address these before making budget decisions.

### Issues Found

| Severity | Platform | Issue | Impact |
|----------|----------|-------|--------|
| HIGH | {platform} | {issue title} | {what it affects} |
| MEDIUM | {platform} | {issue title} | {what it affects} |

### Detail

**{Issue title}** ({severity} — {platform})
{Evidence and explanation. Source: {evidence file, field path}.}
Recommendation: {what to do about it}
```

Sort by severity (HIGH first). If a tracking issue undermines the reliability of a platform's data, note that in the relevant platform findings section too.

### Section 5: Findings by Platform

**One subsection per platform with evidence. Use this structure for each:**

```markdown
## {Platform Name}

### Performance Overview

| Metric | Value | vs. Client Target | vs. Industry Benchmark | Rating | Label |
|--------|-------|-------------------|----------------------|--------|-------|
| Total Spend | {formatted} | — | — | — | OBSERVED |
| Revenue | {formatted} | — | — | — | OBSERVED |
| ROAS | {formatted} | {target}x min | {benchmark range} | {Floor/Healthy/Strong} | CALCULATED |
| CPA | {formatted} | ${break-even} max | ${industry median} | {rating} | CALCULATED |
| {other key metrics} | ... | ... | ... | ... | ... |

Source: {evidence file}, account_overview

### Primary Constraint

**{Title from diagnosis.primary_constraint}**

{Description. Include the evidence trail and why this is the #1 lever.}

(Source: {evidence file}, diagnosis.primary_constraint)

### Key Findings

**{Finding title}** [{label}]
{Evidence and significance.}
(Source: {evidence file}, findings[{index}])

{Repeat for each finding. Group by theme if >5 findings.}

### Anomalies

{If anomalies exist in evidence:}
- **{Description}** [{label}] — {evidence}

### Platform Opportunities

| Priority | Action | Expected Impact | Confidence |
|----------|--------|-----------------|------------|
| HIGH | {action} | {impact} | {high/medium/low} |
| MEDIUM | {action} | {impact} | {confidence} |
```

**Benchmark comparison format throughout:**
`{metric} is {value} — {Floor/Healthy/Strong} (Floor: {x}, Healthy: {y-z}, Strong: {a}+)`

Example: `CPA is $45 — Floor (Floor: >$64, Healthy: $21-$32, Strong: <$19)`

Always show client-specific targets alongside industry benchmarks when available.

### Section 7: Profitability Analysis

```markdown
## Profitability Analysis

{If Shopify data available:}
**Financial source of truth:** Shopify ({date range})

| Metric | Value | Label | Formula |
|--------|-------|-------|---------|
| Total Revenue | {formatted} | OBSERVED | Shopify total revenue |
| Total Ad Spend (all platforms) | {formatted} | CALCULATED | Sum of all platform spend |
| MER | {formatted}x | CALCULATED | ${revenue} / ${spend} = {MER} |
| Blended ROAS | {formatted}x | CALCULATED | ${revenue} / ${paid ad spend} = {ROAS} |
| Blended CPA | {formatted} | CALCULATED | ${spend} / {orders} = ${CPA} |
| AOV | {formatted} | OBSERVED | Shopify AOV |

{If COGS available:}
### Contribution Margin Analysis

| Level | Calculation | Value | % of Revenue |
|-------|------------|-------|--------------|
| Revenue | — | {formatted} | 100% |
| CM1 (Product Margin) | Revenue - COGS | {formatted} | {%} |
| CM2 (Fulfillment Margin) | CM1 - shipping - packaging - fulfillment | {formatted} | {%} |
| CM3 (Marketing-Inclusive) | CM2 - ad spend - processing - returns | {formatted} | {%} |

Break-even CPA: ${AOV} × {margin%} = ${value} [CALCULATED]
Target CPA: ${break-even} × 0.65 = ${value} [CALCULATED]
Minimum ROAS: 1 / {margin%} = {value}x [CALCULATED]
Target ROAS: {min} × 1.4 = {value}x [CALCULATED]

{If COGS NOT available:}
> **Note:** COGS not provided. Using industry estimate of {X}% for {vertical}. All margin calculations below are labeled ASSUMPTION. Actual profitability may differ significantly. [ASSUMPTION]

{If Shopify NOT available:}
> ⚠️ **No Shopify data.** All profitability figures below use platform-reported revenue, which is typically inflated 20-50%. Treat as directional only. [ASSUMPTION]
```

Include the "Good ROAS but bad profit" check results if any flags were triggered.

### Section 10: Tiered Roadmap

```markdown
## Implementation Roadmap

### Week 1 (Days 1-7): Quick Wins + Fixes
- {Action} — {expected result}
- {Action} — {expected result}

### Month 1 (Days 8-30): Optimization
- {Action} — {expected result}
- {Action} — {expected result}

### Months 2-3 (Days 31-90): Strategic Moves
- {Action} — {expected result}
- {Action} — {expected result}
```

**Rules:**
- Week 1 = tracking fixes, budget adjustments, quick config changes. Things that can be done in a day.
- Month 1 = campaign restructuring, creative refreshes, flow optimization. Things that need a full optimization cycle.
- Months 2-3 = new channel launches, incrementality tests, LTV programs. Things that need planning and ramp time.
- Each action references back to an opportunity number from Section 8.

## DOCX Adaptation Notes

When generating DOCX output (user explicitly requested), apply these adjustments:

1. **Read the docx SKILL.md first** for formatting instructions (fonts, styles, spacing).
2. **Add a cover page** with client name, date, "Growth Audit Report", and your agency branding.
3. **Add a table of contents** after the cover page.
4. **Use proper heading hierarchy:** H1 for sections, H2 for subsections, H3 for finding titles.
5. **Tables should be formatted** with alternating row colors and header row styling per docx SKILL.md.
6. **Page breaks** before each major section (Sections 1, 5, 6, 7, 8, 10).
7. **Footer** with page numbers and client name.
8. **The content and structure are identical** — only the formatting wrapper changes.

## Single-Platform Report Adjustments

When only one evidence file exists:

1. **Report title:** "{Client Name} — {Platform} Audit Report" (not "Growth Audit Report")
2. **Omit Section 6** (Cross-Channel Diagnosis) entirely.
3. **Section 7 (Profitability):** Note that MER requires total cross-channel spend and is not calculable from a single platform. Provide platform-specific ROAS and CPA analysis instead.
4. **Section 3 (Missing):** Emphasize that cross-channel insights require additional platform audits.
5. **All other sections remain** — just with content from a single platform.
