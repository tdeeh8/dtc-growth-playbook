# Audit Report Template

This is the canonical structure for all audit reports generated by the synthesizer. The template supports two depth modes (**Quick** and **Full**) to serve different audiences. Follow the section order exactly. Omit sections only when explicitly noted.

**Output format:** Markdown by default. DOCX only if user explicitly requests it.

---

## Report Depth Modes

The synthesizer selects a depth mode before generating. The mode controls how much detail each section includes, NOT which sections appear.

| Mode | When to use | Audience | Target length |
|------|-------------|----------|---------------|
| **Quick** | Default for single-platform audits. Also when user says "quick", "summary", "just the highlights". | Marketers scanning for insights and actions | ~100-150 lines |
| **Full** | Default for multi-platform (3+) audits. Also when user says "full", "detailed", "comprehensive", "deep dive". | Strategists doing deep analysis, or client deliverables needing full audit trail | ~250-350 lines |

**Override:** User can always force either mode regardless of platform count. "Give me a quick version" on a 5-platform audit = Quick. "Give me the full report" on a single platform = Full.

**Key differences by mode** (details in each section below):

| Element | Quick mode | Full mode |
|---------|-----------|-----------|
| Evidence labels (`[OBSERVED]`, `[CALCULATED]`) | Appendix only | Appendix only |
| Source citations (`Source: evidence.json, field`) | Appendix only | Inline |
| Sections 2-4 (Scope, Missing, Tracking) | Collapsed into one callout block | Full sections |
| Performance Overview table | 5-6 key metrics, no Label column | Full metrics with Label column |
| Findings | Top 3-5 only, 2-3 sentences each | All findings, full evidence |
| Campaign/product tables | Top 5 rows + summary row | All rows with meaningful data |
| Brand Analytics / raw data tables | Narrative summary with 2-3 callouts | Table with narrative |
| Profitability Analysis | Break-even table + one paragraph | Full CM waterfall + context |
| Opportunities | Merged into Top 5 Actions | Full prioritized table |
| Formulas | Show once per metric, inline | Show once per metric, inline |
| Dedup rule | Same for both modes (see below) | Same for both modes |

---

## Deduplication Rule (Both Modes)

**First-mention rule:** When a metric or finding appears in the Executive Summary, later sections reference it briefly — do NOT restate the full explanation, calculation, or benchmark comparison. Use a callback like:

- "TACoS at 25.98% (see Executive Summary) drives the ad-dependency problem."
- "The 7pt ACOS cushion noted above means..."

**Benchmark tiers** (Floor/Healthy/Strong) appear ONCE per metric in the report — at the metric's first detailed mention. Later references just state the value and rating without repeating the tier ranges.

**TACoS / ACOS / key metrics:** State the calculation formula once (e.g., "TACoS = $6,290 / $24,210 = 25.98%"). Do not restate the formula in subsequent sections.

## Report Structure

Both modes use the same section order. Quick mode collapses or shortens sections as noted.

### Header Block

```markdown
# {Client Name} — Growth Audit Report

**Date:** {Report generation date}
**Prepared by:** Tanner Hill, {Department context}
**Platforms audited:** {list of platforms with evidence files}
**Date range:** {earliest start date} to {latest end date across all evidence}
**Report type:** {Single-Platform / Cross-Channel Synthesis}
```

---

### Section 1: Executive Summary

**Both modes: identical.** 2-3 paragraphs. No bullet points. Business-owner readable — no jargon.

**Paragraph 1:** State the headline finding. What is the single biggest thing affecting this business's growth? Lead with the answer, not the process. Example: "Kodiak Leather is leaving an estimated $17K-$29K in revenue on the table because its two best-performing campaigns are budget-constrained while exceeding their ROAS targets."

**Paragraph 2:** Supporting context. What other significant findings support or complicate the headline? Mention 2-3 secondary findings. Reference the profitability picture if Shopify data is available.

**Paragraph 3:** What to do about it. Summarize the top 2-3 recommended actions in plain language. Reference the Tiered Roadmap for implementation order.

**Rules:**
- No metrics in the exec summary unless they're essential to the headline (e.g., "3.91x ROAS vs. 3.3x target"). Keep it to 2-3 numbers max.
- Write for a business owner, not a media buyer. "You're paying for clicks that aren't turning into sales" not "CTR/CVR ratio indicates post-click conversion inefficiency."
- Every claim traces to a finding or evidence — but don't cite sources inline here. The rest of the report handles that.
- **This section sets the "first mention" for key metrics.** Later sections should not restate what's already here.

---

### Sections 2-4: Scope, Limitations, and Tracking Health

These three sections handle audit context: what was analyzed, what's missing, and whether tracking issues affect data quality.

#### Quick mode: Collapsed Scope Block

In Quick mode, collapse Sections 2-4 into a single callout block immediately after the Executive Summary. No separate headings.

```markdown
> **Scope:** {Platform(s)} audit, {date range}. AOV tier: {tier}. {One sentence on what's missing and the biggest limitation, e.g., "No Shopify data — profitability figures use Seller Central revenue and are directional only."} {If HIGH-severity tracking issues exist, add: "⚠️ Tracking issue: {one-line description}. Fix before making budget decisions."}
```

**Rules for Quick scope block:**
- Maximum 3 sentences total
- Only mention tracking issues that are HIGH or CRITICAL severity AND affect what the marketer should do (conversion tracking gaps, attribution problems). Omit extraction-method issues (ag-Grid blocked, date picker quirks, account defaulting to wrong entity).
- If zero actionable tracking issues exist, omit the tracking sentence entirely
- No evidence file references — those go in the appendix

#### Full mode: Three Separate Sections

**Section 2: What Was Audited**

```markdown
## What Was Audited

| Platform | Depth | Date Range | Evidence File |
|----------|-------|------------|---------------|
| {platform} | {deep/standard} | {start} – {end} | {filename} |
```

One row per platform with an evidence file. Include the evidence filename for traceability.

If a manifest exists, also note: audit type (full / platform-specific), AOV tier, who requested the audit and any specific focus areas.

**Section 3: Accessible vs. Missing**

```markdown
## What This Report Covers — and What It Can't

**Platforms analyzed:** {list}

**Not included in this audit:**
- {Platform} — {reason: SKIPPED / NOT_STARTED / no access}

**What this means for the analysis:**
- {specific limitation, e.g., "Without Shopify data, profitability figures use platform-reported revenue (typically inflated 20-50%). Treat margin calculations as directional only."}
- {specific limitation}
```

Be explicit about what's missing and how it limits the analysis.

**Section 4: Tracking Health**

**Include ONLY if actionable tracking issues were detected.** "Actionable" means the issue affects data quality or what the marketer should do (conversion tracking gaps, attribution mismatches, EMQ scores below threshold). Extraction-method problems (column virtualization, date picker quirks, account defaults) are NOT tracking health — they go in the Methodology appendix.

If no actionable tracking issues exist across any platform, omit this section entirely.

```markdown
## Tracking Health

{If high-severity issues exist, lead with:}
> ⚠️ **Tracking issues detected.** Performance metrics in this report may be inflated or understated. Address these before making budget decisions.

| Severity | Platform | Issue | Impact | Recommendation |
|----------|----------|-------|--------|----------------|
| HIGH | {platform} | {issue title} | {what it affects} | {what to do} |
```

Sort by severity (HIGH first). Keep to table format — no expanded "Detail" subsection unless the issue is complex enough to warrant explanation.

### Section 5: Findings by Platform

One subsection per platform with evidence. Structure varies by mode.

#### Quick mode: Condensed Platform Section

```markdown
## {Platform Name}

### Key Metrics

| Metric | Value | Rating |
|--------|-------|--------|
| {metric 1} | {formatted} | {Floor/Healthy/Strong or ⚠️/✅} |
| {metric 2} | {formatted} | {rating} |
| {metric 3} | {formatted} | {rating} |
| {metric 4} | {formatted} | {rating} |
| {metric 5} | {formatted} | {rating} |
```

**Quick mode Key Metrics rules:**
- **5-6 metrics maximum.** Use the platform's key metrics shortlist (see below). No Label column — labels go in appendix.
- Show break-even/target comparison in the Rating column (e.g., "⚠️ 2.1pt to break-even") — not a separate column.
- Show benchmark tier on first mention only. Later tables just show the rating.
- Formula for each metric shown once, inline below the table: `TACoS = $6,290 / $24,210 = 25.98%`

**Quick mode findings:** After the metrics table, include:
- **Primary Constraint** — 2-3 sentences. What's the #1 issue?
- **Key Findings** — Top 3-5 findings only. Each is 2-3 sentences max. No `[LABEL]` tags, no `(Source: ...)` citations. Apply the dedup rule — skip findings that just restate metrics from the table.
- **Campaign/Product tables** — Top 5 rows only, sorted by spend or revenue. Include a summary row for "All others" if more exist. Add a "Verdict" or "Cushion" column for quick scanning.
- **Anomalies** — Only anomalies that affect what the marketer should do. Skip extraction-method anomalies.
- **Skip the Platform Opportunities subsection** — those merge into Top 5 Actions (Section 7 in Quick mode).

#### Full mode: Detailed Platform Section

```markdown
## {Platform Name}

### Performance Overview

| Metric | Value | vs. Client Target | vs. Industry Benchmark | Rating |
|--------|-------|-------------------|----------------------|--------|
| Total Spend | {formatted} | — | — | — |
| Revenue | {formatted} | — | — | — |
| ROAS | {formatted} | {target}x min | {benchmark range} | {Floor/Healthy/Strong} |
| CPA | {formatted} | ${break-even} max | ${industry median} | {rating} |
| {other key metrics} | ... | ... | ... | ... |

(Source: {evidence file}, account_overview)

### Primary Constraint

**{Title from diagnosis.primary_constraint}**

{Description. Include the evidence trail and why this is the #1 lever.}

(Source: {evidence file}, diagnosis.primary_constraint)

### Key Findings

**{Finding title}**
{Evidence and significance. 3-5 sentences.}
(Source: {evidence file}, findings[{index}])

{Repeat for each finding. Group by theme if >5 findings. Apply dedup rule — brief callback to exec summary findings, not full restatement.}

### Anomalies

{Only actionable anomalies — skip extraction-method artifacts:}
- **{Description}** — {evidence and what to do about it}

### Platform Opportunities

| Priority | Action | Expected Impact | Confidence |
|----------|--------|-----------------|------------|
| HIGH | {action} | {impact} | {high/medium/low} |
| MEDIUM | {action} | {impact} | {confidence} |
```

**Full mode note:** Even in Full mode, the Label column is removed from tables. Labels and full source citations go in the Methodology appendix. Formulas are shown inline once per metric.

#### Platform Key Metrics Shortlists

When building the metrics table (either mode), lead with these platform-specific metrics. The synthesizer should always include these and can add 1-2 more if highly relevant.

| Platform | Key Metrics (in order) |
|----------|----------------------|
| Google Ads | Total Spend, Revenue, ROAS vs target, CPA vs break-even, Impression Share (Search), Conv Rate |
| Meta Ads | Total Spend, Revenue, ROAS vs target, CPM, Frequency, Thumbstop Rate, Conv Rate |
| Amazon Ads | Total Ad Spend, TACoS, Account ACOS vs break-even, CVR, Ad Dependency Ratio, Featured Offer % |
| Shopify | Total Revenue, AOV, Orders, New vs Returning %, Refund Rate, Discount Rate |
| GA4 | Sessions, Engaged Sessions %, Conv Rate by channel, Revenue by source, Bounce Rate |
| Klaviyo | Revenue Attributed, Flow Revenue %, Campaign Revenue %, List Growth Rate, Unsubscribe Rate |
| BigCommerce | Total Revenue, AOV, Orders, Conv Rate, Abandoned Cart Rate |
| Website/CRO | Page Load Time, Mobile Conv Rate, Bounce Rate, Cart Completion Rate, Checkout Drop-off |

#### Summary Table Rules (Both Modes)

When the evidence contains large data tables (campaigns, ASINs, keywords, search queries, flows, etc.):

**Quick mode:** Show the top 5 rows by the primary sort metric (usually spend or revenue), plus a summary/rollup row for "All others." Use a condensed column set — drop columns that don't help the reader act (e.g., impressions, start date).

**Full mode:** Show all rows with meaningful activity (spend > $0, revenue > $0, etc.). Drop rows with zero activity. Use a complete but not exhaustive column set.

**Both modes:** Roll up to the highest useful level. Per-ASIN tables become product-line summaries with 2-3 individual callouts for outliers. Per-keyword tables become campaign-level summaries unless keyword-level data is the point. The raw data lives in the evidence JSON — the report interprets it.

**Benchmark comparison format (first mention only):**
`{metric} is {value} — {Floor/Healthy/Strong} (Floor: {x}, Healthy: {y-z}, Strong: {a}+)`

Example: `CPA is $45 — Floor (Floor: >$64, Healthy: $21-$32, Strong: <$19)`

Subsequent mentions: `CPA at $45 (Floor)` — no tier range.

### Cross-Channel Diagnosis (Multi-Platform Only)

**OMIT this section entirely for single-platform reports.** This section appears between Findings (Section 5) and Profitability (Section 6) only when 2+ platforms were audited.

```markdown
## Cross-Channel Diagnosis

{Introductory sentence: "With {N} platforms audited, the following cross-channel patterns were detected."}

### {Pattern Name} — {Confidence: HIGH/MEDIUM/LOW}

**What we found:** {Description with specific numbers from specific evidence files.}

**Evidence:**
- {Platform A}: {metric} = {value} (Source: {evidence file}, {field path})
- {Platform B}: {metric} = {value} (Source: {evidence file}, {field path})
- {Calculation or comparison}: {formula} = {result}

**What this means:** {Business implication in plain language.}

**Recommended action:** {What to do about it.}

{Repeat for each detected pattern.}
```

Only include patterns where evidence actually supports the detection. Do NOT include a pattern section just because it's in the library — only when the data shows it.

### Section 6: Profitability Analysis

#### Quick mode: Break-Even Summary

```markdown
## Profitability

{One-line data source note, e.g.:}
> Revenue source: Seller Central ($24,210 YTD). No Shopify data — treat margins as directional.

| Product/Line | Selling Price | All-In Cost | Break-Even ACOS/CPA | Target ACOS/CPA |
|-------------|--------------|-------------|---------------------|-----------------|
| {product 1} | {price} | {cost} | {break-even} | {target} |
| {product 2} | {price} | {cost} | {break-even} | {target} |
| **Blended** | — | — | **{blended BE}** | **{blended target}** |

{One paragraph: Where the account stands relative to break-even. Which campaigns/channels are profitable, which are marginal, which are losing money. Reference the campaign table from Section 5 — don't rebuild it here.}
```

**Quick mode rules:**
- No CM1/CM2/CM3 waterfall — too detailed for quick consumption
- No Label column
- Show formulas once: `Break-even ACOS = (Price - COGS - FBA) / Price`
- If COGS unavailable, state the assumption in one sentence, not a full caveat block
- If a "Good ROAS but bad profit" flag was triggered, call it out in the paragraph

#### Full mode: Complete Profitability Analysis

```markdown
## Profitability Analysis

{If Shopify data available:}
**Financial source of truth:** Shopify ({date range})

| Metric | Value | Formula |
|--------|-------|---------|
| Total Revenue | {formatted} | Shopify total revenue |
| Total Ad Spend (all platforms) | {formatted} | Sum of all platform spend |
| MER | {formatted}x | ${revenue} / ${spend} = {MER} |
| Blended ROAS | {formatted}x | ${revenue} / ${paid ad spend} = {ROAS} |
| Blended CPA | {formatted} | ${spend} / {orders} = ${CPA} |
| AOV | {formatted} | Shopify AOV |

{If COGS available:}
### Contribution Margin Analysis

| Level | Calculation | Value | % of Revenue |
|-------|------------|-------|--------------|
| Revenue | — | {formatted} | 100% |
| CM1 (Product Margin) | Revenue - COGS | {formatted} | {%} |
| CM2 (Fulfillment Margin) | CM1 - shipping - packaging - fulfillment | {formatted} | {%} |
| CM3 (Marketing-Inclusive) | CM2 - ad spend - processing - returns | {formatted} | {%} |

Break-even CPA: ${AOV} × {margin%} = ${value}
Target CPA: ${break-even} × 0.65 = ${value}
Minimum ROAS: 1 / {margin%} = {value}x
Target ROAS: {min} × 1.4 = {value}x

{If COGS NOT available:}
> **Note:** COGS not provided. Using industry estimate of {X}% for {vertical}. Margin calculations are estimates.

{If Shopify NOT available:}
> ⚠️ **No Shopify data.** Profitability figures use platform-reported revenue. Treat as directional only.
```

Full mode includes the "Good ROAS but bad profit" check results and the campaign-by-campaign profitability assessment with cushion analysis.

**Both modes:** No `[CALCULATED]` or `[ASSUMPTION]` tags inline. The Formula column makes provenance self-evident. Data source caveats go in prose, not labels.

### Section 7: Actions and Opportunities

This section combines the prioritized opportunity list and the Top 5 distillation. The two modes handle these differently.

#### Quick mode: Top 5 Actions Only

In Quick mode, skip the full prioritized opportunity table. Go straight to the Top 5.

```markdown
## What to Do — Top 5 Actions

1. **{Action}** — {one-sentence rationale with expected impact}
2. **{Action}** — {rationale}
3. **{Action}** — {rationale}
4. **{Action}** — {rationale}
5. **{Action}** — {rationale}
```

**Quick mode rules:**
- These are the 5 highest-impact, highest-confidence actions from across all platforms
- Each action is one bolded title + one sentence of rationale. No tables, no Evidence column.
- If a CRITICAL tracking fix exists, it takes slot #1 regardless
- Written for a marketer who will scan this section first and maybe only this section

#### Full mode: Prioritized Opportunities + Top 5

```markdown
## Opportunities — Prioritized

### Critical (Fix These First)

| # | Action | Platform | Impact | Confidence | Effort |
|---|--------|----------|--------|------------|--------|
| 1 | {action} | {platform} | {quantified} | {H/M/L} | {Quick/Moderate/Strategic} |

### High Priority

| # | Action | Platform | Impact | Confidence | Effort |
|---|--------|----------|--------|------------|--------|

### Medium Priority
{Meaningful improvements, moderate confidence or effort}

### Low Priority / Exploratory
{Worth investigating but lower impact or confidence}
```

Number opportunities sequentially across all tiers (1, 2, 3... not restarting per tier).

**Then include the Top 5:**

```markdown
## If You Do Nothing Else — Top 5 Actions

1. **{Action}** — {one-sentence rationale with expected impact}
2. **{Action}** — {rationale}
3. **{Action}** — {rationale}
4. **{Action}** — {rationale}
5. **{Action}** — {rationale}
```

Full mode includes both sections. The Top 5 is the distillation — it references opportunity numbers from the table above.

### Section 8: Implementation Roadmap

**Both modes: identical structure.** Quick mode may have fewer items per tier.

```markdown
## Implementation Roadmap

### Week 1 (Days 1-7): Quick Wins + Fixes
- {Action} — {expected result}
- {Action} — {expected result}

### Month 1 (Days 8-30): Optimization
- {Action} — {expected result}
- {Action} — {expected result}

### Months 2-3 (Days 31-90): Strategic Moves
- {Action} — {expected result}
- {Action} — {expected result}
```

**Rules:**
- Week 1 = tracking fixes, budget adjustments, quick config changes. Things that can be done in a day.
- Month 1 = campaign restructuring, creative refreshes, flow optimization. Things that need a full optimization cycle.
- Months 2-3 = new channel launches, incrementality tests, LTV programs. Things that need planning and ramp time.
- Each action references back to an opportunity number from Section 7 (Full mode) or Top 5 item (Quick mode).

### Section 10: Methodology & Sources (Appendix)

**Both modes: always include.** This is where evidence labels, source citations, and extraction notes live — keeping them out of the main report body.

```markdown
---

## Appendix: Methodology & Sources

### Data Sources

| Platform | Source | Date Range | Evidence File |
|----------|--------|------------|---------------|
| {platform} | {specific tool/page} | {range} | `{filename}` |

### Evidence Labels

Every metric in this report has a provenance label tracked in the evidence JSON:

| Label | Meaning |
|-------|---------|
| OBSERVED | Pulled directly from the platform UI or export |
| CALCULATED | Derived from observed values (formula shown in report) |
| INFERENCE | Logical conclusion from observed patterns |
| ASSUMPTION | Estimated where exact data unavailable |
| DATA_NOT_AVAILABLE | Sought but not obtainable |

### Extraction Notes

{Any methodology caveats that don't affect recommendations but matter for reproducibility:}
- {e.g., "Targeting page keyword data not extractable via browser; campaign-level data used instead."}
- {e.g., "Brand Analytics SQR data is weekly snapshot (Week 14), not YTD aggregate."}
- {e.g., "Account defaulted to wrong entity on login; manually switched before extraction."}

### Calculation Reference

{List of all formulas used in this report, each shown once:}
- ACOS = Spend / Sales
- TACoS = Total Ad Spend / Total Revenue (organic + paid)
- Break-even ACOS = (Selling Price - COGS - Fulfillment) / Selling Price
- {etc.}
```

**Rules:**
- This appendix replaces ALL inline `[OBSERVED]`, `[CALCULATED]`, etc. tags throughout the report
- This appendix replaces ALL inline `(Source: evidence.json, field)` citations in Quick mode
- Full mode keeps inline source citations AND has this appendix
- Extraction-method anomalies (date picker issues, column virtualization, account defaults) go here, NOT in Tracking Health or Anomalies

## DOCX Adaptation Notes

When generating DOCX output (user explicitly requested), apply these adjustments:

1. **Read the docx SKILL.md first** for formatting instructions (fonts, styles, spacing).
2. **Add a cover page** with client name, date, "Growth Audit Report", and {Agency} branding (or {Brand} if applicable).
3. **Add a table of contents** after the cover page.
4. **Use proper heading hierarchy:** H1 for sections, H2 for subsections, H3 for finding titles.
5. **Tables should be formatted** with alternating row colors and header row styling per docx SKILL.md.
6. **Page breaks** before each major section (Sections 1, 5, 6, 7, 8).
7. **Footer** with page numbers and client name.
8. **The content and structure are identical** — only the formatting wrapper changes.
9. **Depth mode still applies.** Quick DOCX = shorter doc. Full DOCX = comprehensive doc.

## Single-Platform Report Adjustments

When only one evidence file exists:

1. **Report title:** "{Client Name} — {Platform} Audit Report" (not "Growth Audit Report")
2. **Default depth:** Quick mode (unless user requests full)
3. **Omit Section 5** (Cross-Channel Diagnosis) entirely.
4. **Section 6 (Profitability):** Note that MER requires total cross-channel spend and is not calculable from a single platform. Provide platform-specific ROAS and CPA analysis instead.
5. **Sections 2-4 (Scope):** Use the collapsed scope block (Quick mode behavior even if Full mode is selected for other sections).
6. **All other sections remain** — just with content from a single platform.
